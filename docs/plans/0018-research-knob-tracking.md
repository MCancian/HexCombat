# 0018 — Research Knob Tracking

**Status:** Sketch
**Priority:** Medium (Research visibility)

## The Problem
As our research program scales via `tools/run_sweep.py` and `tools/run_batch.py`, we lose visibility into what combinations of tunable variables (knobs) have been explored. Currently, some variables live in `data/scenario_default.json` (e.g., `feba_base_km`), others in subsystem files like `data/ijfs/ijfs_scenario.json` (e.g., `intel_locked_antiship_strike_bonus`), and parameter sweeps override them dynamically via `DataOverrides`. There is no centralized way to see the full set of combinations we have looked at.

## The Goal
Create an architecture to track, aggregate, and visualize the combinations of variables explored in our research program, allowing the USER and agents to quickly see which parameter spaces have been tested.

## Options for the Implementing Agent
The implementing agent should review these options with the USER before writing code.

### Option A: Promotion to Scenario Variants (The Data-Driven Approach)
Promote all tunable variables currently scattered across `ijfs_scenario.json`, `antiship_crossing_config.json`, etc., into a `"tuning": {}` block within the `scenario` JSON schema. Research runs are then forced to use explicit scenario variants (`data/scenarios/variant_a.json`) rather than runtime `DataOverrides`.
- **Changes required:**
  - Expand `validate_scenario_data.gd` to include a `"tuning"` block.
  - Refactor `IjfsLoaders.gd` and `AntishipLoaders.gd` to accept tuning variables from `GameData.scenario_data` instead of pulling from their own subsystem JSONs.
  - Write a new `tools/make_research_summary.py` script that iterates over `data/scenarios/*.json`, diffs them against `scenario_default.json`, and outputs a Markdown table of all explored combinations.

### Option B: The Ledger (The Runtime Tracking Approach)
Keep the variables where they are, but require the engine to explicitly collect and log them during every game.
- **Changes required:**
  - Add a `GameData.dump_tunables()` method that returns a flat dictionary of known knobs from all loaded/overridden JSONs.
  - Embed this dictionary into the `turn_digests` or the root of the game record generated by `tools/run_selfplay_game.gd`.
  - Write a new `tools/update_research_ledger.py` script that scans all batch records in `reports/batches/` and `reports/sweeps/`, and appends to a central `reports/research_ledger.md` (or CSV) showing exactly what was run.

### Option C: The Unified Tuning Profile (The Split-Architecture Approach)
Separate "Scenario" (laydown, OOB, rules) from "Tuning" (coefficients, bonuses, modifiers).
- **Changes required:**
  - Extract all knobs into a new schema: `data/tuning/tuning_default.json`.
  - Create a `TuningCatalog` autoload alongside `ScenarioCatalog`.
  - Modify `tools/run_batch.py` and `tools/run_selfplay_game.gd` to accept a `--tuning` argument independently of `--scenario`.
  - Research combinations are documented by the matrix of `--scenario` × `--tuning` runs.

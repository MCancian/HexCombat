IJFS → Anti-Ship → Amphibious-Crossing Lethality Knobs

HexCombat (Godot 4 / GDScript) — every tunable in the pipeline that decides how many assault battalions survive the crossing. Values pulled directly from source on 2026-06-28. Reference / recalibration document — no code changed.

Why this exists. In the golden scenario (seed 20260624) the amphibious wave loses ~24 of 36 battalions crossing into theater TO3. The designer's prior is closer to ~25% (~9 of 36). This document maps every knob that drives that outcome so the gap can be tuned with eyes open.

The two-gate structure

Every IJFS effect on a Green anti-ship system passes through two independent gates, each its own random draw:

  1. Gate 1 — Detection (IjfsDetection.gd): is the system found this turn? P(detect). Static/intel-locked targets auto-detect; mobile targets roll against a satellite floor (phase 1) plus an ISR-weighted term (phase 2).
  2. Gate 2 — Strike (IjfsStrike.resolve_strike): given it was found and a munition is paired, is it destroyed, else suppressed? p_destroy then, on survival, p_suppress_if_not_destroyed.

Per-target net effect each turn (one detection draw, then the strike draws only on the detected fraction):

net P(destroy) = P(detect) × p_destroy
net P(suppress) = P(detect) × (1 − p_destroy) × p_suppress_if_not_destroyed
net P(degraded) = net P(destroy) + net P(suppress)
Headline finding. The detection gate is wildly asymmetric across Green anti-ship system classes:
  • Air-launched AShM platforms (moveable): satellite floor = 0.50 when hiding → near-certain detection, and strike p_destroy ≈ 0.55–0.65. Net ~72–75% neutralized (destroyed + suppressed). They get gutted.
  • Mobile coastal AShM launchers (mobile, types 23/24, ~126 launchers): satellite floor = 0.01 when hiding (50× lower) and strike p_destroy ≈ 0.045. Net ~0.3% neutralized. They are barely touched.
Result: the assaulted theater's coastal arsenal fires at near full strength (~283 systems fire in TO3), so the crossing model sinks ~2/3 of the wave. The only lever that degrades the shoot-and-scoot mobile launchers is the C2-suppression fire penalty (×0.70) — and in this seed the IJFS suppresses TO4/TO5 C2, not the assaulted TO3, so the penalty never lands where the wave crosses.

1 · IJFS Detection (Gate 1)

Two phases per IJFS day. Phase 1 (satellite): mobile/moveable targets roll only against the satellite floor. Phase 2 (aircraft ISR): floor + an ISR-weighted detectability term. Static or intel-locked targets are auto-known (P=1.0). Source formula (IjfsDetection.aircraft_detect_target_ids, matches detection_model.final_probability_rule):

P_detect = min(1.0, satellite_floor + base_probability × mobility_mult × posture_mult × weighted_ISR)

3 · C2 suppression lever

The single mechanic that degrades the shoot-and-scoot mobile coastal launchers (which get ~zero direct IJFS suppression). If the IJFS suppresses a theater's C2 node (type 99), every surviving anti-ship system in that TO fires at a reduced fraction this turn. There is no C2-destroy mechanic — suppression models the staff being knocked out.

C2-suppressed-TO gate GameState.resolve_antiship_turn → c2_suppressed_tos per-TO boolean from last_ijfs_writeback.antiship_suppressed_by_type[encode_key(to,99)] boolean Which TOs get the penalty. Depends entirely on which TO's C2 the IJFS chose to suppress. In golden seed: TO4/TO5 suppressed, TO3 (assaulted) NOT — so penalty misses the wave.
This is the crux. The lever exists and is unit-tested (suppressing TO3's C2 strictly reduces TO3 firing), but the IJFS targeting does not prioritize the assaulted TO's C2, so in this seed the only mobile-launcher degrader fires at the wrong theaters. See lever (a) in § Calibration levers.

Worked numeric example — the disparity, side by side

Both targets hiding, IJFS attacking with its best-available anti-ship munition, real numbers plugged in. Net per-turn outcome = P(detect) × p_destroy for destruction, and P(detect) × (1−p_destroy) × p_suppress for suppression.

Mobile Coastal AShM launcher — mobile, hiding (types 23/24)

Gate 1 detect (satellite floor, mobile/hiding) ........ 0.01
  (phase-2 best case: 0.01 + 0.05×1.0×1.0×ISR — stays ~0.02–0.05)
Gate 2 p_destroy (BRE6 pairing 0.045, +mods ≈ −0.005) .. ≈ 0.040
Gate 2 p_suppress_if_not_destroyed (BRE6) ............. 0.22
net P(destroy) = 0.01 × 0.040 = 0.0004
net P(suppress) = 0.01 × 0.96 × 0.22 = 0.0021
net P(degraded) ≈ 0.25%

Even using strike-aircraft (pd 0.14) and a generous phase-2 detect of 0.05, net degraded only reaches ~1%. These launchers fire at essentially full strength. Only the C2 ×0.70 lever touches them — and only if it lands on the assaulted TO.

Air-Launched AShM platform — moveable, hiding (type 3)

Gate 1 detect (satellite floor, moveable/hiding) ..... 0.50
  (phase-2: 0.50 + 0.05×1.2×1.0×ISR → clamps toward ~0.95)
Gate 2 p_destroy (BRE6 0.60, +mod moveable_hiding_air −0.10) ≈ 0.55
Gate 2 p_suppress_if_not_destroyed (BRE6) ............. 0.45
net P(destroy) = 0.95 × 0.55 = 0.52
net P(suppress) = 0.95 × 0.45 × 0.45 = 0.19
net P(degraded) ≈ 71%

Matches the observed ~72% of air-launched platforms destroyed/suppressed. Detection — not the strike table — is the discriminator: a 50× higher satellite floor (0.50 vs 0.01).

The gap in one line: ~71% of air-launched platforms are neutralized each turn vs ~0.25% of mobile coastal launchers. Since the assaulted TO's lethality is dominated by its ~126 untouched mobile coastal launchers (plus patrol boats/CDCMs), the crossing model sees near-full fire and sinks ~2/3 of the wave.

Calibration levers for crossing lethality — ranked

Ranked by how directly each moves the ~67% → ~25% target. Cross-referenced to PLAN.md → Open Questions → "D3-D crossing lethality calibration" candidate levers (a)–(d).

  1. (a) Make the IJFS prioritize the assaulted TO's C2. PLAN lever (a) Highest leverage with the existing machinery: the C2 ×0.70 penalty already exists and is the only degrader of mobile launchers, but in this seed it lands on TO4/TO5, not TO3. Targeting/value-weighting the assault-TO C2 would route the 30% penalty to the wave. Pairs naturally with deepening the penalty (lower C2_SUPPRESSED_FIRE_MULTIPLIER).
  2. Lift the mobile-coastal detection floor and/or strike p_destroy. detection/strike The most physically direct fix to the root cause: satellite_floor_probability.mobile.hiding = 0.01 and pairing p_destroy ≈ 0.045 mean these launchers are nearly invulnerable. Raising the floor (e.g. 0.01 → 0.10–0.20) or the mobile-coastal pd would let the multi-day pre-invasion IJFS actually thin them. Caution: this is a deliberate "shoot-and-scoot" survivability model — verify against intended doctrine. (Note mobile_target_destroy_caps is dormant, so it won't fight you — but the pairing pd values already bake the caps in.)
  3. (b) Tune DEFAULT_ANTISHIP_FIRE_PCT and per-type range_tier. PLAN lever (b) 100% fire-all + neighboring-tier mobile launchers means one TO's whole arsenal (plus adjacent TOs') engages every wave. Dropping fire % (e.g. 100 → 40–60) or tightening mobile-coastal range_tier to own_to cuts missile volume broadly. Blunt but reliable; affects all classes, not just the over-performing ones.
  4. Lower the crossing leak rate — interception / terminal defense. crossing §5 Fewer missiles leak to transports ⇒ fewer BN losses, without touching IJFS. Raise escort_interception.success_prob (currently 0.35–0.55) or terminal_defense.base_probability (0.45), or shrink missile_group_size (4) so escorts spend fewer attempts per missile. Risk: decouples lethality from the IJFS suppression story (transports just get tougher).
  5. (c) Cross-turn magazine gating. PLAN lever (c) Currently dormant (magazine = null, rebuilt full each turn). Persisting AntishipMagazine across turns would let multi-day firing deplete missiles (block_ii_mobile 400, HF store 660) so later waves face fewer shots. Medium leverage, more plumbing; mostly matters across multiple crossing turns, less on turn-1.

Discrepancies, dormant knobs & HexCombat-vs-TIV notes

  • Dormant: mobile_target_destroy_caps (ijfs_scenario.json). Present and detailed (active/hiding caps + ship overrides 0.80–0.95), but not consumed: IjfsStrike.destruction_probability only reaches _legacy_cap_probability when strike_probability_modifiers is empty, and it is non-empty. The low mobile pd values come straight from the pairing table. If you ever clear the modifier list, these caps would suddenly activate — worth knowing before tuning.
  • Dormant: firing-side magazine (AntishipMagazine / antiship_magazine_defaults.json). Fully implemented but wired with magazine = null in resolve_antiship_turn (per-turn rebuild never binds). Documented follow-up = lever (c).
  • Documented divergence: count-based vs per-hull crossing stages. AntishipCrossing ports TIV's count-based interception + terminal-defense stages, not TIV's live per-hull variants that deplete each escort's hq10/hhq9 magazines (services.ship_ammo) and scale defense by damage status (ship_readiness_policy). HexCombat has no ship-ammo/readiness subsystem, so escort interception is bounded only by attempts/success_prob, never by magazine depletion. Per PLAN.md D3-B3 this matches all test_antiship_crossing.py assertions (escort ammo never binds there). It does mean escort screens here never "run dry" across a long crossing — a latent lethality-reducer that exists in TIV but not yet here.
  • RNG model. HexCombat mirrors TIV's source formulas and draw order via an injected Dice, not Python's PRNG bitstream (per AGENTS.md). Values are equivalent in distribution, not byte-identical to a TIV run.
  • maneuver_casualties writeback still empty. The IJFS only writes anti-ship destroyed/suppressed by encode_key(to,type); the ground-casualty half is unimplemented (no shared battalion_id between the IJFS target set and the OOB). Not a crossing-lethality knob, but it means IJFS effects on the assault are entirely mediated through the anti-ship firing plan, with no direct ground attrition path.
  • No value marked "NOT FOUND." Every knob above was read from source. The only "verify" caveats are the two dormant tables (caps, firing magazine) — present in data but not on the live code path in the current wiring.

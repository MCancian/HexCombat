Hex Grid & Geometry

Human-readable summary. Machine reference: docs/systems/hex-grid.md. Part of the TIV port-fidelity audit (Area 1 of 10).

What this system does

Defines the hex coordinate system and the runtime geometry the game runs on: which hexes are adjacent, how far apart two hexes are, shortest paths, reachable ranges, and how each hex's geographic outline is drawn on screen. The grid itself (466 hexes covering Taiwan) is generated offline by the original Python toolchain and shipped as a static file (data/taiwan_hex_grid.json); HexCombat only consumes it.

How faithful is the port?

The map data, projection, and the general shape of the geometry layer port cleanly. The audit found one serious mismatch in the coordinate system itself — now fixed:

✅ Fixed (2026-06-29) — adjacency now uses the correct coordinate system.

The grid file stores offset (odd-r) coordinates — odd rows are nudged half a hex sideways. The original game computes neighbors with row-parity-aware offsets that respect that. HexCombat had been treating the stored numbers as plain axial coordinates and applying a fixed neighbor pattern to every hex.

Measured against true geography (great-circle distance between hex centers) over the 308 interior hexes that have six real ~10 km neighbors:

  • Original odd-r scheme (TIV oracle): 308 / 308 correct
  • Old HexCombat axial scheme: 23 / 308 correct — adjacency wrong on ~92% of interior hexes

Because distance, pathfinding, and reachable-range all build on neighbors, they inherited the error — and adjacency drives movement, combat, FEBA, and the front line, so this was foundational.

Resolution: HexMath now picks the neighbor offsets by row parity (odd-r) and converts offset→cube for distance, matching the oracle. The deterministic golden combat result was re-baselined as a consequence (casualties=2, feba=0.76 → casualties=3, feba=-0.55), and the full test gate passes.

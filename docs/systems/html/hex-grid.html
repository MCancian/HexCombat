Hex Grid & Geometry

Human-readable summary. Machine reference: docs/systems/hex-grid.md. Part of the TIV port-fidelity audit (Area 1 of 10).

What this system does

Defines the hex coordinate system and the runtime geometry the game runs on: which hexes are adjacent, how far apart two hexes are, shortest paths, reachable ranges, and how each hex's geographic outline is drawn on screen. The grid itself (455 hexes covering Taiwan) is generated offline by the original Python toolchain and shipped as a static file (data/taiwan_hex_grid.json); HexCombat only consumes it.

How faithful is the port?

The map data, projection, and the general shape of the geometry layer port cleanly. But the audit found one serious mismatch in the coordinate system itself:

🔴 Confirmed discrepancy — adjacency uses the wrong coordinate system.

The grid file stores offset (odd-r) coordinates — odd rows are nudged half a hex sideways. The original game computes neighbors with row-parity-aware offsets that respect that. HexCombat instead treats the stored numbers as plain axial coordinates and applies a fixed neighbor pattern to every hex.

Measured against true geography (great-circle distance between hex centers) over the 308 interior hexes that have six real ~10 km neighbors:

  • Original odd-r scheme: 308 / 308 correct
  • HexCombat axial scheme: 23 / 308 correct

So on ~92% of interior hexes HexCombat treats a wrong hex as adjacent and misses a true neighbor. Because distance, pathfinding, and reachable-range all build on this, they inherit the error. Adjacency drives movement, combat, FEBA, and the front line — so this is foundational.

Status: flagged in /DECISIONS.md for your call, not auto-fixed — the fix changes the deterministic golden combat result the test suite is pinned to, so it needs a conscious re-baseline. Recommended fix: switch neighbors to odd-r and convert offset→cube for distance.

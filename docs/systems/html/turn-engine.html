Turn Engine & Data Services

Human-readable summary. Machine reference: docs/systems/turn-engine.md. Port-fidelity audit, Area 8 of 10.

What this is

The runtime spine of the game — four always-loaded services: GameData (loads all the static data once: hexes, orders of battle, ships, beaches, theaters), GameState (the turn clock and everything that changes turn to turn), EventBus (signals so parts of the game can react to each other), and Dice (a seeded random-number source so every game is reproducible).

How a turn runs (WeGo)

HexCombat uses a "WeGo" model: both sides write their orders during planning, then the turn resolves all at once in a fixed pipeline:

  1. Air/missile fires (IJFS)
  2. Anti-ship & mines vs the crossing fleet
  3. Amphibious offload (survivors land)
  4. Movement & commitment (both sides)
  5. Ground combat at every contested hex
  6. Front-line retreats & hex ownership
  7. Supply consumption
  8. Cleanup + victory census

A single play_turn(red, green, dice) call drives all of this and returns a structured result — the entry point used by the headless LLM/AI play.

✅ A deliberate adaptation of TIV — not a line-by-line port (correctly so).

The original is a Flask/SQLite app where each phase is a web view advanced by clicking through the UI. HexCombat keeps the same phase order but rebuilds the engine around Godot autoloads, adds a genuine plan-then-resolve order layer, a structured turn result for AI play, and — importantly — a seeded dice abstraction so runs are reproducible (the original used unseeded Python randomness). A validator (validate_no_global_rng) enforces that no game logic reaches for global randomness. These are improvements over the source, by design.

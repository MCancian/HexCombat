View Layer — Rendering, UI & Input

Human-readable summary. Machine reference: docs/systems/view-layer.md. Port-fidelity audit, Area 10 of 10.

What this is

Everything the player sees and clicks: the hex map, unit markers with NATO symbols, the info and composition panels, and the click-to-select-then-move/commit flow. It draws and routes input but holds no game logic — it asks GameState to do things and reads GameData to display them, communicating through the signal bus.

How it works

PieceDoes
HexMapProjects Taiwan's lat/lon hex grid to screen, colors hexes by owner, draws brigade markers + reachable-hex highlights, turns clicks into hex selections.
GameControllerThe glue: click → select brigade → (move if reachable / commit) → end turn; wires signals to GameState.
InfoPanel / CompositionPanelShow the selected hex/brigade details, and buttons for eligible commitments.
SymbolLibraryLoads NATO unit symbols (11 types) from a JSON map; a validator checks every unit type has one.
HexCombat-original — nothing to port here. The original tool was a Flask/HTML/JS web app with Leaflet map overlays; this Godot view was built from scratch, so there's no line-by-line comparison.

Still to come (graphics track): anti-ship/mine visualization, the front-line drawing tool, unit/HUD polish (HP bars, combat popups), and map/terrain polish. These need to be checked by eye in the running game, so they're outside the headless test gate.

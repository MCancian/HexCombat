Ground Combat (BOOTS)

Human-readable summary. Machine reference: docs/systems/ground-combat.md. Port-fidelity audit, Area 2 of 10.

What this system does

When Red and Green brigades end a turn in the same hex, this resolves the fight: it adds up each side's combat strength (maneuver battalions + fire support, times a terrain factor for the defender), forms a force ratio, rolls for casualties on each side, and shifts the front line (FEBA) toward the winner. Casualties are removed, the front moves, hex ownership updates.

✅ Core math is a faithful, near-line-for-line port of TIV's boots_calculator.py → resolve_map_attack: identical loss-rate formulas and clamps, identical fire-support multipliers, identical FEBA math, the same "at least one casualty" rule, and the same rule that artillery battalions can't be picked as casualties.

The combat formula

attacker_strength = Σ(unit strength × supply) + support
defender_strength = (Σ + support) × terrain_modifier  (terrain ≥ 1.0)
ratio = attacker_strength / defender_strength

attacker_loss_rate = clamp(0.20 − (ratio−1)×0.08 + (roll−50)/1000, 0.05 … 0.45)
defender_loss_rate = clamp(0.20 + (ratio−1)×0.10 + (roll−50)/1000, 0.05 … 0.50)
losses = round(unit_count × loss_rate)

balance = (atk − def) / (atk + def)
feba_shift_km = feba_base_km × clamp(balance×2, −2 … 2) × (0.75 + roll/100×0.5)

Where the port differs from TIV

🔴 1. Unit strengths differ on 12 of 17 battalion types.

Verified by running TIV's own calculator: TIV's type→strength lookup never connects the full battalion names the data actually uses, so almost every maneuver unit collapses to strength 1.0 in TIV. HexCombat instead uses a differentiated table (Armor 2.0, Combined Arms 1.5, Amphibious 1.2, Recon 0.7, …). HexCombat clearly ported TIV's intended table rather than its buggy runtime — arguably the right call, but it means HexCombat combat ≠ TIV combat output. One outlier goes the other way: helicopters are 0.5 in HexCombat vs 1.4 in TIV's mapping. Flagged in /DECISIONS.md — recommend keeping HexCombat's table and ratifying it as an intentional divergence.

🔴 2. Front-line movement scale: 2.0 vs 3.5.

HexCombat uses feba_base_km = 2.0; TIV loads 3.5 from config. The front moves ~57% as far per fight in HexCombat — likely the port not carrying the config value. Flagged in /DECISIONS.md (recommend making it scenario-configurable, default 3.5).

⚠ 3. Different random-number generator.

TIV uses NumPy's RNG; HexCombat uses its own seeded dice. The sequence of rolls matches, but the algorithms differ, so HexCombat can't reproduce TIV's exact per-seed numbers — it is internally deterministic only. This is inherent to the engine port, not a bug.

⚠ 4. Minor: combat-detail shape.

HexCombat's per-combat detail dict uses the key support_breakdown and omits support_unit_count; TIV has support_power_breakdown + the count. Cosmetic unless an external consumer expects TIV's exact keys.

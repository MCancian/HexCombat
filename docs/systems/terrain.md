# Terrain — data model, assignment, movement & combat effects, rendering

## 1. Purpose

Gives each of the 466 hexes in `data/taiwan_hex_grid.json` a terrain class (Track F). Terrain
feeds three systems: ground movement cost (weighted Dijkstra + impassable mountains), ground
combat (defender modifier), and the map view (fill tint). Built from real elevation (DEM) and
land-cover rasters, not hand-authored per hex.

## 2. Data model

**`data/terrain/terrain_types.json`** — 5 terrain classes, each with:

| Field | Meaning |
|---|---|
| `defender_modifier` | Multiplies defender strength in ground combat |
| `move_cost` | Movement-point cost to ENTER a hex of this class |
| `impassable` | If true, ground movement can never select the hex as destination or waypoint |
| `color` | Hex fill tint (`#RRGGBB`), used by the view layer |

Values (USER design calls 2026-07-09):

| Class | `defender_modifier` | `move_cost` | `impassable` | `color` |
|---|---|---|---|---|
| plains | 1.0 | 1 | false | `#DCE8C0` |
| hills | 1.5 | 2 | false | `#C4A96A` |
| urban | 2.0 | 1 | false | `#9E9E9E` |
| metropolis | 3.0 | 2 | false | `#6E6E6E` |
| mountain | 2.0 | 1 | true | `#8B7355` |

**`data/terrain/hex_terrain.json`** — `{hex_id: class_name}` for every grid hex; generated by
`tools/terrain/classify_hexes.py`, description field says "edit `overrides.json`, not this file".

**`data/terrain/hex_terrain_stats.json`** — raw per-hex `mean_elev_m`, `elev_std_m`,
`builtup_frac` that classification was computed from (debugging/audit artifact, not consumed at
runtime).

**`data/terrain/hex_land_frac.json`** — per-hex fraction of area over GSHHG land, written by the
grid-reconciliation pipeline (§3); provenance record, not consumed at runtime.

**Loader — `GameData.load_terrain()`** (`scripts/GameData.gd:252-318`): reads
`terrain_types.json` into `terrain_types: Dictionary` (class name → `TerrainType` resource,
`scripts/model/TerrainType.gd`), then reads `hex_terrain.json` and stamps `hex_lookup[hex_id].terrain`
per hex. Fails loud (`push_error` + early return) on: a missing required field, an
unrecognized class referenced by a hex, an unclassified grid hex, or a classified hex not in the
grid — every hex must resolve to exactly one of the 5 classes before the game can run.
`GameData.get_terrain(hex_id) -> TerrainType` (`scripts/GameData.gd:321-324`) is the read API;
returns `null` for an unknown hex_id.

## 3. Assignment pipeline

Two independent Python pipelines under `tools/terrain/`, run against raster/vector caches in
`~/geodata/hexcombat/` (not checked into the repo):

**Grid reconciliation — `tools/terrain/reconcile_grid.py`** (stage 3, commit `bf75d9f`): fits the
existing odd-r lattice formula, computes each hex's land fraction against the GSHHG full-resolution
coastline shapefile (`land_frac`, `reconcile_grid.py:97-104`), and:
1. **Inclusion rule:** keep an existing hex if `land_frac >= 0.05` (`_LAND_FRAC_THRESHOLD`,
   `reconcile_grid.py:46`); drop it otherwise. Scan every lattice cell not already in the grid and
   add it as a candidate if it also clears `land_frac >= 0.05`.
2. **Contiguity filter** (`filter_contiguous`, `reconcile_grid.py:150-170`): flood-fills odd-r
   adjacency from the kept hexes over the union of (kept ∪ candidate) coordinates; a candidate
   survives only if reachable from the kept set. This rejects off-theater land that clears the
   land-fraction threshold but isn't part of the Taiwan landmass (6 mainland-China/Matsu-coast
   candidates rejected in the 2026-07-09 run).
3. Writes the new `data/taiwan_hex_grid.json` (404 of 455 prior hexes kept byte-identical, 51
   dropped, 62 added — net 466), `data/terrain/hex_land_frac.json`, and a plain-text report at
   `tools/terrain/out/reconcile_report.txt`.

Consequences of the regen, all part of commit `bf75d9f`:
- `data/beaches.json` gained a canonical `hex_id` per beach (the grid hex containing the beach's
  lat/lon point); `BeachDef` + loader + validator enforce its presence.
- **Beach 3 (Zhuwei) re-pointed**: `red_ship_reserve[...].beach_hex` `hex_44_14` → `hex_43_14` in
  `data/scenario_default.json` — `hex_44_14` came back pure sea under the reconciled coastline,
  reversing an earlier fix (commit `5fc675f`, which had moved the landing the other direction).
  BDE-99's defender placement moved inland with it, `hex_43_14` → `hex_42_15`, preserving the
  red-lands-coastal / green-defends-adjacent pattern.
- Victory census for `roc_full_defense` (the 451-hex main-island hex list,
  `tools/gen_main_island_hexes.py` / `tools/validate_victory_hexes.gd`) regenerated 451 → 462: 4
  offshore hexes newly qualify under the reconciled coastline (`hex_4_17`, `hex_4_18`, `hex_5_17`,
  `hex_12_17`) and are excluded from that scenario's census the same way Green/Orchid Island
  always were — main-island scope, not the full 466-hex grid.

**Terrain classification — `tools/terrain/classify_hexes.py`** (stages 2 and 4b, commits
`54cf5a9`, `3e5f5c2`): for each hex polygon, samples a DEM raster for `mean_elev_m`/`elev_std_m`
(`hex_elevation_stats`) and a land-cover raster for `builtup_frac` (fraction of pixels in the
built-up class, `hex_builtup_frac`), then classifies (`classify_hex`, lines 117-126):

```
mean_elev_m >= 1000              -> mountain
builtup_frac >= 0.50             -> metropolis   (added stage 4b)
builtup_frac >= 0.20             -> urban
200 <= mean_elev_m < 1000        -> hills
else                             -> plains
```

`tools/terrain/overrides.json` force-sets specific hexes' class post-classification
(`apply_overrides`); edit that file to correct a misclassification, never `hex_terrain.json`
directly. **Metropolis** (stage 4b, USER call: 5th class, `builtup_frac >= 0.50` →
`defender_modifier` 3.0, `move_cost` 2) reclassified exactly 9 hexes out of `urban`: the Taipei
(`hex_42_15`, `hex_42_17`, `hex_43_17`), Taichung (`hex_31_8`, `hex_32_9`), Tainan (`hex_16_4`,
`hex_17_4`), and Kaohsiung (`hex_11_5`, `hex_12_5`) metro cores.

Current class counts (466 hexes): mountain 131 / hills 141 / urban 54 / metropolis 9 / plains 131.

**Preview** — `tools/terrain/preview_terrain.py` renders a static PNG of the classified grid
(`tools/terrain/out/terrain_preview.png`) for human sign-off before committing a classification
change; not part of the runtime or the gate.

## 4. Movement effects

`GameData` threads terrain into `HexMath`'s pathfinding via two callables passed at every
`find_path`/`find_reachable` call site (`scripts/GameData.gd:343-352`):

- **Entry cost** — `_terrain_entry_cost(hex_id)` (`GameData.gd:369-373`) returns
  `get_terrain(hex_id).move_cost`, defaulting to 1 for an unclassified hex (load-time validation
  already guarantees every grid hex is classified; this is a defensive fallback for call sites
  that might run before `load_terrain()`).
- **Impassability** — `_with_impassable(blocked)` (`GameData.gd:357-363`) merges the caller's
  explicit `blocked` list with every hex whose `TerrainType.impassable` is true (mountains today)
  before calling into `HexMath`.

`HexMath.find_path` (`scripts/HexMath.gd:63-96`) is a plain cost-aware Dijkstra shortest path —
no min-one-step guarantee. `HexMath.find_reachable` (`scripts/HexMath.gd:102-137`) computes the
Dijkstra reachable set (cumulative entry cost ≤ `max_distance`) **plus** every passable direct
neighbor of the start hex regardless of its cost — the min-one-step guarantee: a unit that has not
yet moved this turn may always take one step into an adjacent passable hex even if that hex's
`move_cost` alone exceeds its remaining movement points. Mountain hexes are never in the reachable
or path set regardless of cost or distance.

Tests: `tests/terrain_movement_test.gd` — `test_slow_brigade_min_one_step_and_distance_limits`,
`test_fast_brigade_reaches_plains_at_two_but_not_hills_then_plains_chain`,
`test_no_mountain_hex_ever_reachable`, `test_find_path_routes_around_mountains`.

## 5. Combat effects

`CombatResolver.resolve_at` (`scripts/resolvers/CombatResolver.gd:38-65`, pure — no `GameData`
access) takes `defender_terrain_modifier: float = 1.0` as its last parameter and forwards it
straight into `CombatCalculator.resolve_map_attack`. `GameState._defender_combat_modifier(hex_id)`
(`scripts/GameState.gd:561-568`) computes the value passed in:

```gdscript
func _defender_combat_modifier(hex_id: String) -> float:
    var terrain := GameData.get_terrain(hex_id)
    var terrain_modifier: float = terrain.defender_modifier if terrain != null else 1.0
    return terrain_modifier * 1.0
```

The `* 1.0` is a **deliberate seam**, not dead code: a future situational multiplier (the
backlogged beach first-landing ×2 defender penalty) plugs in there without touching
`CombatResolver` or `GameData`. Terrain resolves at `hex_id` — the *defended* hex — regardless of
which side started there (`GameState.gd:585-587`), called from `_resolve_combat_at`
(`GameState.gd:581-597`).

Per-class defender modifier table: plains ×1.0, hills ×1.5, urban ×2.0, mountain ×2.0
(mountain is impassable to movement but can still be a combat modifier if reached by other means —
currently moot since nothing can enter it), metropolis ×3.0.

**Golden invariant:** seed `20260624` → whole-game `casualties=6, feba=-3.04` (was
`casualties=3, feba=-0.96` before the modifier activated) — byte-stable gate pin
(`tools/validate_headless_turn.gd`; `tools/validate_cleanup.gd`'s single-contested-hex
`EXPECTED_COMBAT_FINGERPRINT` keeps `casualties=3` but also moves to `feba=-3.04`). The golden
scenario's contested hex is `hex_43_16` (urban, ×2.0) — activating the modifier (stage 5, commit
`4bc9e6c`) flipped that fight from "Attacker Advantage" (ratio 1.5, feba +1.337, attacker_losses 1,
defender_losses 1) to "Defender Advantage" (ratio 0.75, feba -0.955, attacker_losses 2,
defender_losses 0) in `docs/examples/llm_result_after_turn.json` — the FEBA line retreats, exactly
the "stronger defender" shape expected. Full re-baseline rationale: `PLAN.md` → Decisions,
2026-07-09 "Track F: activate the defender terrain modifier..." entry.

Tests: `tests/combat_terrain_modifier_test.gd` — `test_modifier_1_0_no_terrain_bonus`,
`test_modifier_2_0_urban_defender_bonus`, `test_modifier_3_0_metropolis_defender_bonus` (call
`CombatResolver.resolve_at` directly at a fixed 20-vs-5 mismatch, hand-verified against the
loss-rate/FEBA formulas).

## 6. Observation surfacing

`LLMGameAPI._occupied_hex_observations()` (`scripts/LLMGameAPI.gd:198-215`) adds a `"terrain"` key
(the class name string, `""` if unclassified) to every `occupied_hexes` entry — documented in
`docs/LLM_OBSERVATION_SCHEMA.md`'s Occupied hex object table. No JSON-schema change was needed:
`occupied_hexes` is an untyped array in `schemas/llm_observation.schema.json`.

## 7. Rendering

`HexMap.get_hex_color(hex_id)` (`scripts/HexMap.gd:351-363`) tints the hex fill by
`TerrainType.color`, then lerps the ownership color over it at weight 0.35
(`terrain_color.lerp(ownership_color, 0.35)` — 65% terrain / 35% ownership, terrain-dominant) for
owned/contested hexes, or toward mid-gray at weight 0.15 for unowned hexes. Unclassified hexes
(empty `TerrainType.color`) fall back to ownership-only fill — the pre-Track-F behavior. The 0.35
weight is a **USER call** made by comparing rendered screenshots at 0.65 vs 0.35 ownership weight
(`reports/terrain_view.png` vs `reports/terrain_view_035.png`); terrain-dominant won because the
projector needs the terrain class to read at a glance, with ownership still visible as a color
cast. Full rendering detail (beach glyphs, brigade markers, projection, input) lives in
`docs/systems/view-layer.md` §3.
